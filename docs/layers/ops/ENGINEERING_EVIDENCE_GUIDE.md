# 기술 경험 기록 가이드

기술 경험 기록은 개발자로서 설명할 수 있는 문제 해결, 성능 개선, 품질 개선, 기술 의사결정을 모으는 공간입니다. 단순 작업 로그와 다릅니다. 작업 로그가 “무엇을 했는가”를 남긴다면, 기술 경험 기록은 “왜 어려웠고, 어떻게 판단했으며, 어떤 근거로 개선됐는가”를 남깁니다.

현재 Notion 정본은 `개발자 기술 경험 DB`입니다. Codex, Notion, GitHub PR, 문서 운영 과정에서 생긴 에이전트/도구 운영 사고는 `에이전트 운영 로그 DB`에 분리해 남깁니다.

## Notion 정본 위치

| 공간 | 용도 | Data source |
| --- | --- | --- |
| 개발자 기술 경험 | 제품 개발/운영 문제해결, 성능개선, 품질개선, 기술결정 | `collection://7f052514-c585-4621-ad28-b54bb2eac5a8` |
| 에이전트 운영 로그 | Codex, Notion, GitHub PR, 문서 운영 사고와 재발 방지 | `collection://8646042e-8ea0-4dd5-a056-c01a8ec096ec` |

## 기록 종류

| 종류 | 기록 대상 | 예시 |
| --- | --- | --- |
| `문제해결` | 개발/운영 중 막힌 문제를 원인까지 파고 해결한 기록 | CI 실패 원인 분석, Docker 기동 실패 해결, API 장애 격리 |
| `성능개선` | 전후 수치가 있는 개선 기록 | 쿼리 시간 단축, batch 처리량 증가, 테스트 시간 감소 |
| `품질개선` | 안정성, 테스트, 검증, 에러 처리, 운영 안전성 개선 | retry/backoff 보강, validation 강화, idempotency 추가 |
| `기술결정` | 여러 선택지 중 왜 이 구조를 택했는지 남기는 기록 | Kafka 제외, pipeline/backend 책임 분리, DB schema 선택 |

트러블슈팅은 `개발자 기술 경험 DB` 안의 `문제해결` 유형입니다. 모든 기술 경험이 트러블슈팅은 아니며, 성능 개선과 품질 개선은 반드시 전후 근거를 함께 남깁니다.

## 포트폴리오 후보

| 값 | 의미 |
| --- | --- |
| `대표` | 면접이나 자기소개서의 메인 사례로 쓸 만한 강한 경험 |
| `보조` | 필요하면 언급할 수 있지만 주력 사례는 아닌 경험 |
| `기록` | 프로젝트 운영상 필요하지만 포트폴리오 소재는 아닌 기록 |

단순 GitHub/Notion/PR/문서 운영 문제는 개발자 기술 경험이 아니라 `에이전트 운영 로그 DB`에 둡니다. 다만 CI, Docker, 외부 API, 인증, 환경 변수처럼 제품 개발/운영에 영향을 주는 문제를 실제로 조사하고 해결했다면 `개발자 기술 경험 DB`의 `문제해결` 후보로 올릴 수 있습니다.

## 대표 사례 기준

- 시스템 영향이나 사용자 가치가 분명합니다.
- 원인을 직접 조사해 좁혀간 과정이 있습니다.
- 해결 전후 검증이 있습니다.
- 가능하면 정량 지표나 재발 방지 장치가 있습니다.
- 면접에서 “왜 그 선택을 했는가”와 “다시 한다면 무엇을 바꿀 것인가”를 설명할 수 있습니다.

## 성능/품질 개선 기록 필드

| 필드 | 작성 기준 |
| --- | --- |
| `측정 지표` | p95 응답시간, batch 처리 시간, 쿼리 실행 시간, 테스트 시간, 실패율 등 |
| `개선 전` | 변경 전 수치나 관찰 상태 |
| `개선 후` | 변경 후 수치나 관찰 상태 |
| `개선율` | 퍼센트, 배수, 감소 시간처럼 비교 가능한 표현 |
| `측정 방법` | 어떤 명령, 로그, benchmark, 수동 검증으로 측정했는지 |
| `근거 링크` | PR, CI log, benchmark 결과, 관련 문서 |

수치가 없는 품질 개선도 남길 수 있지만, “테스트가 늘었다”보다 “어떤 실패를 막기 위해 어떤 검증을 추가했는가”를 씁니다.

## 과장 금지 원칙

- 억지로 성능 개선을 만들지 않습니다.
- 추정 원인을 확정 원인처럼 쓰지 않습니다.
- 명령어만 쓰지 않고, 어떤 판단과 검증을 했는지 씁니다.
- 작은 운영 문제를 대표 포트폴리오 사례로 부풀리지 않습니다.
- 외부 서비스 제약, 임시 우회, 남은 리스크를 숨기지 않습니다.
- 개인정보, 토큰, 쿠키, 원문 데이터, 계정 정보는 기록하지 않습니다.
