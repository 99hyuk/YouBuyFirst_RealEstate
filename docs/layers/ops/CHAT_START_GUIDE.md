# 채팅 시작과 작업 영역 선택 가이드

새 채팅에서 긴 역할 프롬프트를 반복하지 않기 위한 시작 규칙입니다. 이미 대화에 주입된 긴 문서는 다시 전문 출력하지 않고, 필요한 줄만 좁게 확인합니다.

## 기본 원칙

- 사용자가 `front 작업`, `ops 컨텍스트 압축 모드`, `crawl 다음 작업 봐줘`처럼 legacy 이름으로 말하면 `ui`, `ops`, `community` 같은 primary work area로 해석합니다.
- 작업 영역이나 범위 없이 `뭐 해야 해?`, `이제 뭐하지?`라고 물으면 바로 구현하지 않고 가까운 작업 영역과 다음 작업 후보를 보여줍니다.
- 시작할 때 작업 영역, 수정 대상, 기록 위치, 주요 위험을 한두 문장으로 선언합니다.
- 작업 영역이 겹치면 구현보다 계약, 기획 확인, `ops` 조율을 먼저 제안합니다.
- 제품 방향, 공개 배포 리스크, 작업 영역 경계, 검증 가능성, 문서 보존 원칙과 충돌하면 먼저 질문하거나 반박합니다.
- 스킬, gstack, Notion, 세션 로그는 시작 루틴이 아닙니다. 실제 작업 성격이 필요할 때만 읽거나 실행합니다.
- 새 작업을 시작하기 전에는 전체 히스토리 대신 최근 open PR과 현재 세션 PR만 확인해, 늦게 도착한 사람 리뷰나 Codex 자동 리뷰의 actionable 의견이 남아 있는지 봅니다.

## 범위 없는 질문

사용자가 작업 범위 없이 묻는 경우:

1. 프로젝트가 domain/layer 작업 영역으로 나뉜다고 설명합니다.
2. 각 작업 영역을 한 줄로 보여줍니다.
3. `docs/current/HANDOFF.md`의 가까운 후보 중 1-2개를 추천합니다.
4. 사용자가 작업 영역을 고르게 묻습니다.

```text
지금은 작업 영역을 먼저 고르면 좋아요.

- community: 커뮤니티 글 수집과 source policy
- stock: 종목 master, symbol, alias, 종목 식별
- indicator: 반응 방향, 집계, 개미 심리 지수
- market: 시세/호가, quote cache
- simulation: 가상 계좌, 주문, 체결, 원장
- agent: AI 판단, 커뮤니티별 성과 비교
- ui: 대시보드와 화면
- ops: 기획, 문서, Notion, PR/CI

가까운 후보는 <후보 1> 또는 <후보 2>입니다. 어느 작업 영역으로 갈까요?
```

## 짧은 작업 영역 지시

짧은 지시만으로 충분합니다.

```text
front 작업해
crawl 쪽 맡아줘
ops로 Notion 정리해
agent 영역에서 커뮤니티별 성과 비교 설계해
프론트 와이어프레임 잡아줘
```

에이전트는 루트 `AGENTS.md`의 라우팅 표와 담당 도메인/layer `AGENTS.md`를 적용합니다. 긴 역할 프롬프트를 새로 만들지 않습니다. legacy 지시는 `docs/layers/ops/WORK_AREAS.md`의 alias 표로 해석합니다.

`front` 지시를 받으면 `docs/layers/ui/README.md`의 시작 지시를 함께 적용합니다. 기본값은 Vue 3 + Vite + TypeScript 기반 저충실도 와이어프레임이며, 최종 브랜드 디자인은 별도 합의 전에는 고정하지 않습니다.

## 시작 선언

```text
작업 영역: ui / 수정 대상: 대시보드 화면 / 기록 위치: PR + Notion 작업 로그 / 주요 위험: API 계약 미확정으로 mock 데이터 경계 유지.
```

오류로 끊긴 채팅을 이어받을 때:

```text
작업 영역: ops / 수정 대상: 채팅 안정성 조사와 문서 보정 / 기록 위치: PR + 필요 시 에이전트 운영 로그 / 주요 위험: 세션 로그와 Notion fetch 출력 과다.
컨텍스트 압축 모드로 시작해, 이미 주입된 긴 지시는 다시 출력하지 않고 필요한 문서만 좁게 확인하겠습니다.
```

## 반박과 확인 게이트

아래 상황은 바로 수용하지 말고 위험, 문제, 대안 1-2개, 사용자 결정 지점을 짧게 말합니다.

- 법적/운영 리스크가 있는 크롤링, 시세 재배포, 개인정보, 원문 노출
- 여러 작업 영역이 동시에 영향을 받는 DB schema, API contract, Notion 구조 변경
- PR 크기가 커지거나 무관한 변경이 섞이는 작업
- 제품 방향과 다른 화면/기능 추가
- 문서 원칙과 충돌하는 요청
- 검증할 수 없는 완료 주장

```text
이 요청은 그대로 진행하면 <리스크>가 있습니다. 저는 <대안 A> 또는 <대안 B>가 더 안전하다고 봅니다. 어느 쪽으로 갈까요?
```

## 기획 확인이 필요한 경우

- 작업 영역 경계를 바꾸는 경우
- 공개 배포 정책, 크롤링 정책, 법적 리스크에 영향을 주는 경우
- DB schema나 API contract가 여러 작업 영역에 영향을 주는 경우
- 화면 정보 구조나 핵심 지표 정의가 제품 방향을 바꾸는 경우

```text
이 화면은 ui 작업이지만 지표 정의가 indicator와 연결됩니다. mock 화면은 진행할 수 있고, 실제 API 연결은 indicator 계약 확인 후 별도 PR로 나누겠습니다.
```

## Notion 책임

각 작업 영역은 자기 작업의 사실 기록을 남기고, `ops`는 사람이 보는 프로젝트 지도와 기획 정합성을 관리합니다.

| 영역 | 책임자 | 갱신 시점 |
| --- | --- | --- |
| 작업 로그 DB | 각 작업 영역 | PR 생성/수정/머지 |
| 다음 작업 DB | 각 작업 영역 + ops | 후보나 우선순위 변경 |
| 개발자 기술 경험 DB | 문제를 겪은 작업 영역 | 제품 개발/운영 문제, 개선, 기술 결정 |
| 에이전트 운영 로그 DB | 사고를 낸 에이전트 또는 ops | Codex/Notion/GitHub/PR/문서 운영 문제 |
| 루트 대시보드, 제품 기획 | ops | 작업 영역 상태, 범위, 지표 정의 변경 |

## 응답 틀

```text
작업 영역: <work-area> / 수정 대상: <target> / 기록 위치: <records> / 주요 위험: <risk>.
먼저 관련 문서를 필요한 범위만 확인하고, 이 작업 영역 안에서 진행하겠습니다.
```

```text
이 작업은 <work-area A>와 <work-area B>가 겹칩니다. <이유> 때문에 바로 구현하면 충돌 가능성이 있습니다. 먼저 <확인할 내용>을 정리하겠습니다.
```
