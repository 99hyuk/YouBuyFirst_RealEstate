# 제품 기획, 크롤링 리스크, 병렬 작업 트랙 재정리 설계

## 배경

너나사 (YouBuyFirst)의 초기 MVP는 네이버 종토방과 에펨코리아에서 글을 수집하고 종목별 감성을 저장하는 데이터 파이프라인입니다. 사용자는 이후 최종 제품 기획에 다음 내용을 반영하고 싶다고 요청했습니다.

- 커뮤니티별 수익률 비교 에이전트 추가
- 일반 게시판형 소스와 종목 게시판형 소스의 수집 방식 분리
- 공개 배포 시 크롤링 리스크 사례 기반 검토
- 여러 채팅/에이전트가 병렬 작업할 수 있는 작업 트랙 분리
- 실시간 또는 지연 시세를 투자 참고 사이트의 핵심 요소로 격상

## 설계 원칙

- 30분 커뮤니티 집계는 제품 핵심으로 유지합니다.
- 공개 UI는 원문보다 집계 지표, 대표 키워드, 감성 비율, AI 재서술 근거를 중심으로 보여줍니다.
- 원문 재게시, 작성자 추적, 닉네임 랭킹, 프로필 수집은 하지 않습니다.
- 수집 행위 자체의 리스크는 남으므로 소스별 활성화 상태를 둡니다.
- 실시간 시세는 이용 조건을 확인한 뒤 사용하고, 필요하면 지연 시세나 사용자 개인 API 연결로 대체합니다.
- 실제 MSA로 바로 분리하지 않고, 한 저장소 안에서 문서/파일/PR 소유권을 강하게 나눕니다.

## 제품 변경

최종 기획에 `커뮤니티별 수익률 비교 에이전트`를 추가합니다. 이 에이전트는 각 커뮤니티의 감성 신호를 기반으로 가상 전략을 만들고, 신호 발생 후 1시간, 6시간, 24시간, 3일, 7일 수익률을 비교합니다.

예시 전략:

- 에펨코리아 추종 전략
- 디시 주식갤러리 역추종 전략
- 네이버 종토방 공포매수 전략
- 토스 종목 커뮤니티 보유자 심리 전략

표현은 투자 권유가 아니라 사후 검증 리포트로 제한합니다.

## 수집 전략

커뮤니티는 두 종류로 나눕니다.

일반 게시판형:

- 에펨코리아 주식 게시판, 디시 주식갤러리
- 최근 글 목록을 수집하고, 글 안에서 종목을 인식합니다.

종목 게시판형:

- 네이버 종토방, 토스 종목 커뮤니티
- 모든 종목을 30분마다 전수 수집하지 않습니다.
- `CrawlTarget` 큐로 종목별 우선순위를 관리합니다.
- 우선순위는 시가총액, 거래대금, 전일 급등락, 일반 게시판 언급 증가, 사용자 관심 종목, 최근 활동량을 반영합니다.

## 공개 배포 정책

소스별 상태:

- `enabled`: 운영 정책 검토 후 공개 환경에서 수집 가능
- `public-demo-only`: 공개 화면에는 fixture, 지연 데이터, 샘플 데이터만 사용
- `local-research-only`: 로컬 연구와 시연용으로만 실행
- `disabled`: 약관, 접근 제한, 안정성 문제로 비활성화

공개 UI 정책:

- 원문을 재게시하지 않습니다.
- 닉네임, 프로필, 작성자별 추적을 노출하지 않습니다.
- 작성자 표시명은 hash로 제한합니다.
- 제한 원문은 30일 보관을 기본으로 합니다.
- 근거는 언급량 변화, 감성 비율, 대표 키워드, 링크 수, AI 재서술 요약으로 제공합니다.

## 리스크 사례

`docs/LEGAL_RISK_CASES.md`에 다음 사례를 정리합니다.

- 야놀자 vs 여기어때: 형사 무죄와 민사 손해배상 리스크가 갈린 사례
- 잡코리아 vs 사람인: 데이터베이스 제작자 권리 침해가 인정된 사례
- 공개된 개인정보 수집 사건: 공개 정보라도 목적/범위/침해 정도를 비교해야 하는 사례
- Meta/X vs Bright Data: 공개 데이터 수집이 방어 가능한 경우도 있지만 소송 리스크가 남는 사례
- Clearview AI: 개인 식별/생체정보 대량 수집 리스크 사례

## 병렬 작업 트랙

네 개 트랙을 둡니다.

- `community-data-platform`: 커뮤니티 수집, 소스 어댑터, 종목별 수집 타깃, 수집 정책
- `signal-intelligence`: 종목 인식, 감성 분석, 열기 지수, 커뮤니티별 수익률 비교
- `market-simulation-engine`: 시세/호가, Redis quote cache, 모의투자, AI 에이전트
- `product-ops-experience`: 대시보드, 관리자 경험, 문서, Notion, PR/CI, 배포 정책

각 트랙은 `docs/workstreams/` 아래 독립 문서를 갖습니다. 다른 채팅에서 작업할 때는 `AGENTS.md`, `docs/CURRENT_HANDOFF.md`, `docs/FINAL_PRODUCT_PLAN.md`, 담당 트랙 문서를 먼저 읽습니다.

## 검증

이 변경은 문서 전용입니다. 검증은 다음 기준으로 수행합니다.

- 문서 링크와 경로가 실제 파일과 맞는지 확인
- `git diff --check`로 공백 오류가 없는지 확인
- 변경된 Markdown 파일에서 제목 구조와 깨진 링크 후보를 확인
- PR 본문에 변경 범위와 후속 작업을 한국어로 기록
