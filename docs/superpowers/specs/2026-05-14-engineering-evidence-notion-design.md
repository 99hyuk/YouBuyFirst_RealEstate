# Engineering Evidence Notion Design

## 배경

너나사 (YouBuyFirst)의 Notion은 지금까지 작업 로그, 트러블슈팅, 다음 작업을 기록하는 운영 허브 역할을 했습니다. 다만 기존 `트러블슈팅 DB`는 개발자 포트폴리오 관점의 문제 해결 기록이라기보다 GitHub, Notion, Docker 같은 도구 운영 이슈까지 섞인 사고 기록에 가까워졌습니다.

사용자의 목표는 단순 작업 기록이 아니라 개발자로서 설명할 수 있는 기술 경험을 쌓는 것입니다. 따라서 문제 해결, 성능 개선, 품질 개선, 기술 의사결정을 한 구조에서 모으되, 포트폴리오 후보와 일반 운영 기록을 구분할 수 있어야 합니다.

## 결정

기존 `트러블슈팅 DB`의 목적을 확장해 `기술 경험 기록 DB`로 재정의합니다.

새 DB를 여러 개 만드는 대신 하나의 DB에 `종류`와 `포트폴리오 후보` 속성을 둡니다. 이렇게 하면 나중에 면접이나 자기소개서에 쓸 수 있는 강한 사례만 필터링하면서도, 일상적인 운영 이슈와 기술 결정도 잃지 않습니다.

## 기록 종류

| 종류 | 의미 | 예시 |
| --- | --- | --- |
| `문제해결` | 개발/운영 중 막힌 문제를 원인까지 파고 해결한 기록 | CI 실패, Docker 기동 실패, API 장애 격리 |
| `성능개선` | 전후 수치가 있는 개선 기록 | 쿼리 시간 단축, batch 처리량 증가, 테스트 시간 감소 |
| `품질개선` | 안정성, 테스트, 검증, 에러 처리, 운영 안전성 개선 | retry/backoff, validation 강화, idempotency 보강 |
| `기술결정` | 여러 선택지 중 왜 이 구조를 택했는지 남기는 기록 | Kafka 제외, pipeline/backend 책임 분리 |
| `도구/운영` | GitHub, Notion, 문서, CI 운영 과정에서 생긴 문제 | PR 본문 인코딩, Notion schema rename 이슈 |

`도구/운영`도 기록하지만, 기본적으로 포트폴리오 대표 사례로 밀지는 않습니다. 이 기록들은 프로젝트 운영 성숙도를 보여주는 보조 근거로 봅니다.

## 포트폴리오 후보 구분

`포트폴리오 후보` 속성을 둡니다.

| 값 | 의미 |
| --- | --- |
| `대표` | 면접/자기소개서에서 메인 사례로 쓸 만한 강한 경험 |
| `보조` | 필요하면 언급할 수 있지만 주력 사례는 아닌 경험 |
| `기록` | 프로젝트 운영상 필요하지만 포트폴리오 소재는 아닌 기록 |

대표 사례는 다음 조건을 만족해야 합니다.

- 문제의 비즈니스/시스템 영향이 분명합니다.
- 원인을 직접 조사해 좁혀간 과정이 있습니다.
- 해결 전후 검증이 있습니다.
- 가능하면 정량 지표나 재발 방지 장치가 있습니다.

## 성능/품질 개선 속성

성능 개선과 품질 개선을 나중에 따로 뽑을 수 있도록 아래 속성을 추가합니다.

| 속성 | 용도 |
| --- | --- |
| `측정 지표` | p95 응답시간, batch 처리 시간, 쿼리 실행 시간, 테스트 시간 등 |
| `개선 전` | 변경 전 수치 |
| `개선 후` | 변경 후 수치 |
| `개선율` | 퍼센트 또는 배수 |
| `측정 방법` | 어떤 명령, 로그, benchmark로 측정했는지 |
| `근거 링크` | PR, CI log, benchmark 결과, 관련 문서 |

지금 당장 억지로 성능 개선을 만들지는 않습니다. 실제 병목이나 반복 작업이 보일 때 기준을 먼저 세우고, 측정 가능한 방식으로 개선합니다.

## 루트 대시보드 구조

루트 페이지는 전체 DB를 길게 보여주지 않습니다. "현재 상태를 빠르게 보는 command center"로 유지합니다.

루트에는 아래만 노출합니다.

- 오늘의 상태
- 트랙 지도
- 최근 작업 3-5개
- 다음 작업 3-5개
- 최근 기술 경험 3-5개
- 전체 DB로 이동하는 링크
- 제품 기획과 작업 맥락 페이지 링크

전체 작업 로그, 전체 기술 경험 기록, 전체 다음 작업은 각각 별도 페이지나 DB view에서 확인합니다.

## 제품 기획과 작업 맥락 페이지

Notion에 `제품 기획과 작업 맥락` 페이지를 둡니다. 이 페이지는 사용자가 현재 프로젝트를 볼 때 "지금 무엇을 만들고 있고, 왜 이 순서로 작업하는지"를 이해하기 위한 문서입니다.

구성:

- 제품 한 줄 정의
- 현재 MVP 목표
- 최종 제품 방향
- 현재 우선순위
- 보류 중인 것
- Codex가 이해한 핵심 판단 기준
- 병렬 트랙별 현재 상태
- 다음 의사결정 후보

이 페이지는 긴 기획서가 아니라 요약 지도입니다. 자세한 내용은 `docs/FINAL_PRODUCT_PLAN.md`, `docs/PROJECT_BRIEF.md`, 작업 로그, 트랙 문서로 연결합니다.

## Codex 협업 원칙

사용자는 무조건적인 수용보다 토론을 통한 발전을 원합니다. 따라서 `AGENTS.md`와 Notion 운영 기준에 아래 원칙을 반영합니다.

- Codex는 사용자의 요구를 일단 존중하되, 모순, 리스크, 더 나은 대안이 보이면 질문하거나 반박합니다.
- 반박은 막기 위한 반박이 아니라 제품 품질, 개발자 경험, 포트폴리오 가치, 운영 리스크를 높이기 위한 것입니다.
- 사용자가 명시적으로 빠른 실행을 원하지 않는 한, 기획/구조 변경은 2-3가지 선택지와 추천안을 먼저 제시합니다.

## Plugin 사용 원칙

Superpowers와 gstack은 목적이 있을 때 사용합니다.

- Superpowers는 기획, 설계, 구현 계획, 검증, 디버깅 같은 작업 흐름의 게이트로 사용합니다.
- gstack은 브라우저 QA, 시각적 확인, 성능/품질 검증처럼 실제 확인 가치가 있을 때 사용합니다.
- 플러그인 사용 자체를 성과로 보지 않습니다. 기록에는 어떤 도구를 썼는지보다 어떤 판단과 검증을 했는지를 남깁니다.

## 구현 범위

다음 구현 작업에서 처리할 내용:

- Notion `트러블슈팅 DB`를 `기술 경험 기록 DB`로 재구성합니다.
- DB 속성에 `종류`, `포트폴리오 후보`, 성능/품질 개선 지표 필드를 추가합니다.
- 기존 트러블슈팅 기록은 `도구/운영` 또는 `문제해결`로 초기 분류합니다.
- 루트 대시보드의 전체 DB 노출을 최근 3-5개 중심 구조로 바꾸고 전체 DB 링크를 분리합니다.
- `제품 기획과 작업 맥락` 페이지를 만듭니다.
- 저장소 문서에 새 기록 기준과 협업 원칙을 반영합니다.

## 제외 범위

- 실제 성능 개선 작업을 억지로 만들지 않습니다.
- 기존 작업 로그 DB를 삭제하지 않습니다.
- 포트폴리오 문장이나 자기소개서 초안은 이번 범위에 포함하지 않습니다.
- Notion 자동화나 GitHub Actions 연동은 이번 범위에 포함하지 않습니다.

## 검증 기준

- Notion 루트 페이지에서 최근 기록만 보이고 전체 DB는 링크로 이동할 수 있습니다.
- 기술 경험 기록 DB에서 `종류`와 `포트폴리오 후보`로 필터링할 수 있습니다.
- 성능/품질 개선 기록에 전후 수치와 측정 방법을 남길 수 있습니다.
- `AGENTS.md`, `docs/WORKFLOW.md`, `docs/TROUBLESHOOTING_GUIDE.md`가 새 기준을 설명합니다.
- 변경 PR 본문과 Notion 작업 로그가 같은 카드형 구조를 따릅니다.
