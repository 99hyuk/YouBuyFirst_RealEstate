# 제품/기술 고민 메모

이 문서는 사용자가 작업 중 던진 제품/기술 고민을 나중에 다시 찾기 위한 얇은 기록입니다. 확정된 제품 방향은 `docs/FINAL_PRODUCT_PLAN.md`, 현재 작업 기준은 `docs/CURRENT_HANDOFF.md`, 기술/운영 리스크는 `docs/TECHNICAL_RISK_REGISTER.md`에 반영합니다.

중요한 기술 의사결정이나 취업 포트폴리오 소재는 Notion 개발자 기술 경험 DB에 별도 기록합니다. 이 문서는 Notion을 대체하지 않고, 대화 중 생긴 고민을 잃어버리지 않기 위한 인덱스입니다.

## 기록 규칙

- 사용자가 "고민거리", "나중에 볼 수 있게", "이거 기록"처럼 말하면 이 문서에 짧게 추가합니다.
- 한 항목은 배경, 현재 판단, 남은 확인으로 나눕니다.
- 이미 결론이 난 항목도 삭제하지 않고 `현재 판단`을 갱신합니다.
- 세부 근거가 길어지면 관련 기획/리스크/작업 단위 문서로 승격합니다.

## 현재 고민 목록

### 2026-05-20. 커뮤니티 기반 개미 심리 지수를 둘지

- 배경: 제품 핵심이 커뮤니티 반응 지표이므로, 사용자가 종목별 분위기를 한눈에 이해할 대표 점수가 필요했습니다. 기존에는 `반응 방향`, `열기 지수`, sentiment bar가 흩어져 있었습니다.
- 현재 판단: `개미 심리 지수`를 추가합니다. 0-100 점수로 종목/기간별 낙관, 비관, 과열, 무관심 정도를 보여주되, 매수/매도 신호가 아니라 커뮤니티 반응의 관찰 지표로 둡니다. 산식 후보는 언급량 변화, 낙관/비관 반응 비율, 표현 강도, 인기글/개념글 확산, 소스 다양성, 표본 수 신뢰도, 시세 지연 상태를 함께 봅니다.
- 남은 확인: 산식 가중치, 30분/1시간/24시간 window, 소스별 가중치, 화면 점수 구간 문구, 낮은 표본 수에서 점수를 숨길 기준을 `data`와 `front` 트랙에서 확정합니다.

### 2026-05-19. 공개 화면에 지연 시세를 직접 표시할지

- 배경: 차트는 TradingView 같은 외부 위젯으로 처리하더라도, 종목 카드와 상세 화면에는 현재가/등락률이 보여야 투자 참고 서비스로 납득된다는 판단이 있었습니다.
- 현재 판단: MVP와 포트폴리오 단계에서는 정식 재배포 라이선스를 먼저 구매하지 않습니다. 기본 조합은 `yfinance` + FinanceDataReader로 정합니다. `yfinance`는 국내/미국 시세와 거래량의 1차 provider로 두고, FinanceDataReader는 국내 종목 메타데이터, 일봉/스냅샷 보강, 국내 수급 후보 provider로 둡니다. `pykrx`는 기본 조합에서 빼고, FinanceDataReader로 부족한 KRX 수급 검증이 필요할 때만 보조 후보로 남깁니다. 다만 이것은 약관 리스크가 사라졌다는 뜻이 아니라, 호출량이 작고 서비스 검증 단계인 동안 provider를 분리해 리스크를 감수하는 선택입니다. 공개 화면에는 제한된 quote snapshot만 보여주고, 원시 분봉/호가/대량 OHLC를 공개 데이터 서비스처럼 제공하지 않습니다.
- 보강: `.KS`/`.KQ` 한국 Yahoo Finance 데이터는 원천 20분 지연으로 보고, pipeline은 기본 10분 주기로 quote snapshot을 갱신합니다. 그래서 공개 화면의 국장 quote는 최대 30분 지연으로 표시합니다. 미국 quote는 지연 시간을 단정하지 않고 10분 주기 refresh snapshot으로 표시합니다. 수급 데이터는 장중 실시간이 아니라 전 거래일 기준 개인/외국인/기관 흐름으로 분리합니다.
- 남은 확인: `yfinance`와 FinanceDataReader 각각의 공개 표출 조건, 국장/미장 지연 기준, quote snapshot 보존 기간, 화면별 표시 범위, 포트폴리오 공개 배포 전 법적/운영 리스크 리뷰를 확인합니다. 실제 트래픽이 커지거나 서비스화 단계에 들어가면 국내는 KRX/KOSCOM, 미국은 public display 권한이 있는 데이터 벤더 계약으로 전환할지 다시 판단합니다.

### 2026-05-18. 프론트와 크롤링/데이터 개발을 병행할지

- 배경: 프론트에서 기획과 화면 구성이 계속 발견되면서 작업이 끝없이 늘어나는 느낌이 있음.
- 현재 판단: 병행하는 편이 맞습니다. 다만 새 기능을 계속 여는 방식이 아니라, 현재 대시보드/종목 상세/커뮤니티 화면을 기준으로 API 필드와 데이터 계약을 역으로 뽑고 네이버 종토방 + 에펨코리아 수집부터 세로로 연결합니다.
- 남은 확인: 프론트 fixture에서 실제 API contract 후보를 뽑고, crawl/data/backend가 먼저 채울 수 있는 최소 응답을 정합니다.

### 2026-05-18. RAG와 벡터DB를 어디에 쓸지

- 배경: 크롤링한 글과 뉴스/리포트/이벤트를 AI 판단에 활용할 때 RAG나 벡터DB가 필요한지 고민함.
- 현재 판단: MVP의 3줄 요약은 SQL 집계, 대표 키워드, 이벤트 타임라인만으로 가능합니다. RAG/벡터DB는 후순위로 두고, 과거 유사 이벤트 검색, 사용자 질문형 분석, 이슈 클러스터링에 적용하는 편이 자연스럽습니다.
- 남은 확인: `StockEvent`, `CommunityPost` snippet, news/report metadata 중 어떤 단위를 embedding 후보로 둘지 후속 설계에서 정합니다.

### 2026-05-18. 종목 은어와 별칭을 어떻게 처리할지

- 배경: 커뮤니티에는 `삼전`, `하닉`, `엔비`, 밈성 표현처럼 종목을 부르는 은어가 많음.
- 현재 판단: 벡터DB를 실시간 종목 판정의 정본으로 쓰지 않습니다. 먼저 alias registry, 종목 마스터, 문맥 규칙으로 후보를 좁히고 애매한 경우만 LLM 검증을 둡니다. 벡터/LLM은 새 별칭 후보 발굴 보조로 쓸 수 있습니다.
- 남은 확인: alias 후보의 승인 기준, confidence 낮은 후보의 저장 여부, 오탐 방지 테스트를 정합니다.

### 2026-05-18. NoSQL, Redis, 채팅/게시판을 넣을지

- 배경: 사용자 체류와 재미를 위해 게시판이나 실시간 채팅을 넣을지 고민함.
- 현재 판단: NoSQL은 필수는 아닙니다. 채팅을 하더라도 초기에는 MySQL에 메시지를 저장하고 Redis는 pub/sub, 접속 상태, typing, rate limit, 안 읽은 수 캐시 같은 실시간 보조 역할로 쓰는 편이 좋습니다. 자유 게시판보다 종목별 반응방이 제품과 더 잘 맞습니다.
- 남은 확인: 사용자 생성 반응은 외부 커뮤니티 반응 지표와 분리할지, 신고/차단/삭제/금지어 정책을 어디까지 구현할지 정합니다.

### 2026-05-18. 커뮤니티별 수익률 비교를 현실적으로 구현할 수 있는지

- 배경: 일반 게시판형과 종목 게시판형 커뮤니티의 반응을 이후 가격 흐름과 비교하려는 기획이 현실적인지 고민함.
- 현재 판단: "예측 시스템"이 아니라 "사후 검증 실험"으로 만들면 현실적입니다. 일반 게시판형은 글에서 종목을 인식하고, 종목 게시판형은 게시판 자체가 종목 컨텍스트가 됩니다. 신호 발생 후 1시간, 6시간, 24시간, 3일, 7일 forward return을 비교합니다.
- 남은 확인: 최초 MVP는 네이버 종토방 + 에펨코리아로 좁히고, 표본 수 기준과 추종/역추종 strategy version을 명확히 둡니다.

### 2026-05-18. 추가 일반 게시판형 주식 커뮤니티 후보

- 배경: 에펨코리아 주식, 뽐뿌 증권포럼, 디시 미주갤, 디시 주식갤/국내주식 계열 외에 종목별 방이 아닌 일반 글 작성형 후보가 더 있는지 확인함.
- 현재 판단: 후보는 더 있습니다. 1차는 에펨코리아 주식, 디시 미주갤, 디시 주식갤/국내주식 계열, 뽐뿌 증권포럼입니다. 2차는 클리앙 주식한당, 팍스넷 커뮤니티/전체글/강추이종목, 딴지 주식클럽, 네이버 카페형 미국주식 커뮤니티입니다. 3차는 오늘의유머 경제, 개드립 주식/재테크, 이토랜드 주식, MLB파크 불펜 내 주식 글처럼 대형 커뮤니티 안의 산발적 주식 글입니다. 다만 공개 접근성, robots/약관, 게시글 품질, 광고성/홍보성, 종목 인식 난이도를 먼저 검토해야 합니다.
- 남은 확인: source registry 후보로만 두고, 바로 adapter 구현하지 않습니다. 공개 배포 전에는 법적/운영 리스크와 원문 노출 정책을 확인합니다.

### 2026-05-18. 부동산 버티컬을 어떻게 둘지

- 배경: SSAFY 관통프로젝트 주제가 부동산/음식/여행이라, 너나사와 병행 가능한 확장 주제를 고민함.
- 현재 판단: 주식 MVP 안에 섞지 않고 후순위 별도 버티컬로 둡니다. 인벤처럼 주식 사이트와 부동산 사이트가 상단 스위치로 이동하되, 도메인 모델은 분리합니다.
- 남은 확인: 주식 MVP가 안정된 뒤 지역/단지, 실거래가, 전세/청약/정책 키워드 중심으로 별도 기획합니다.

### 2026-05-18. 종목 한줄평 표현 수위

- 배경: 분석 화면에서 "이거 매수하는 건 돈에 대한 모독임"처럼 강한 한줄평을 써도 되는지 고민함.
- 현재 판단: 금융 사이트처럼 `매수/매도 흐름`, `순매수`, `호가 잔량`, `매수 언급/매도 언급`을 보여주는 것은 관찰 데이터라서 가능성이 높습니다. 문제는 단어 자체가 아니라 서비스가 특정 행동을 평가하거나 유도하는 말투입니다. 직접 `사라/팔아라`가 아니더라도 `매수하는 건 돈에 대한 모독`은 특정 종목의 매수 행위를 서비스가 평가하는 말처럼 읽힐 수 있습니다. 따라서 재미있는 톤 자체를 금지하기보다, 서비스 결론과 커뮤니티 톤을 분리합니다. 서비스 voice는 `커뮤니티 반응: 강한 우려 우세`, `매수 언급보다 매도 언급 증가`, `과열/손실 우려 표현 다수`, `조롱 섞인 부정 반응 증가`처럼 관찰형으로 씁니다. 짠맛/팩트폭격 한줄평은 살리되, 농담의 대상은 사용자가 아니라 커뮤니티 반응과 실제 지표의 괴리로 둡니다. 예: `커뮤니티 온도는 90도, 7일 수익률은 냉장고`, `기대감은 상한가, 근거는 아직 개장 전`.
- 남은 확인: 화면 문구에서 `사라`, `팔아라`, `수익 보장`, 조롱형 단정 표현을 서비스 voice로 쓰지 않는지 점검합니다. `매수/매도` 단어는 흐름 데이터인지 행동 평가인지 구분해 리뷰합니다. 과격한 문구를 쓰려면 원문 인용 또는 `커뮤니티 톤`으로 라벨링하고, 표본 수와 출처를 함께 표시합니다.

### 2026-05-18. 주식 커뮤니티 우선순위 산정 기준

- 배경: 추가 일반 게시판형 주식 커뮤니티를 객관적으로 많이 쓰는 순서대로 정하고 싶음.
- 현재 판단: 커뮤니티별 공개 MAU가 거의 없으므로 완전한 객관 순위는 어렵습니다. 우선순위는 전체 사이트 트래픽, 주식 게시판 특화성, 게시글 빈도, 공개 접근성, 수집 정책 리스크, 광고성/홍보성 노이즈를 함께 봅니다.
- 남은 확인: adapter 구현 전 각 후보의 robots/약관, 목록 공개 여부, HTML 안정성, 최신 글 수, 조회/댓글/추천 필드 존재 여부를 별도 source registry 후보로 검증합니다.

### 2026-05-18. 종목 상세 팩트폭격 헤드라인

- 배경: 종목 상세에 들어갔을 때 이미지 예시처럼 강한 한줄평을 보여주고 싶음. 이 문구는 커뮤니티 분석이 아니라 실제 시황과 종목 상태 분석에 기반해야 함.
- 현재 판단: 가능하고, 제품 차별점이 될 수 있습니다. 다만 `커뮤니티 반응 한줄평`과 분리해 `종목 상태 한줄평`으로 설계합니다. 근거는 가격/추세, 기술 지표, 재무/성장, 밸류에이션, 뉴스/공시, 애널리스트 컨센서스에서 가져오고, 최소 2개 이상의 근거 묶음이 같은 방향일 때만 강한 카피를 씁니다. 사용자가 준 `종목 고르다 이거 골랐으면 안목을 의심해봐`, `분할매수 한다고 했는데 전부 다 물린 거 맞지?`, `이거 매수하는 건 돈에 대한 모독임`은 `roast` 레퍼런스로 보관합니다.
- 남은 확인: front fixture와 backend/API 명세에 `headlineTone`, `headline`, `subtitle`, `evidence`, `asOf`, `dataQuality`, `personalizedSafe` 후보 필드를 두고, `roast` 단계 카피는 공개 배포 전 별도 리뷰합니다. 검은 네모 배너 UI와 레퍼런스 이미지는 `docs/STOCK_DETAIL_COPY_GUIDE.md`를 따릅니다.

### 2026-05-18. 프론트 화면별 기획과 하위 상세 화면 관리
