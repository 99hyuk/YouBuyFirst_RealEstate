# 채팅 시작과 트랙 선택 가이드

이 문서는 새 채팅이나 병렬 작업 채팅을 시작할 때 에이전트가 어떻게 반응해야 하는지 정리합니다. 사용자는 긴 역할 프롬프트를 반복하지 않아도 됩니다.

## 기본 원칙

- 사용자가 트랙을 말하면 에이전트가 역할 프롬프트를 자동으로 확장합니다.
- 사용자가 트랙을 말하지 않고 `뭐 해야 해?`, `이제 뭐하지?`, `다음 작업 뭐야?`처럼 물으면 바로 구현하지 않습니다.
- 빈 채팅이나 범위가 없는 요청에서는 먼저 트랙을 설명하고, 사용자가 선택하도록 돕습니다.
- 에이전트는 시작할 때 작업 트랙, 수정 대상, 기록 위치, 주요 위험을 짧게 선언합니다.
- 트랙이 겹치면 구현보다 계약, 기획 확인, 또는 `ops` 조율을 먼저 제안합니다.

## 빈 채팅에서의 응답 원칙

사용자가 새 채팅에서 작업 범위 없이 `뭐 해야 해?`라고 물으면 아래 순서로 답합니다.

1. 현재 프로젝트가 일곱 트랙으로 나뉜다고 설명합니다.
2. 각 트랙을 한 줄로 보여줍니다.
3. `CURRENT_HANDOFF.md`의 가까운 다음 작업 후보를 바탕으로 추천 1-2개를 제안합니다.
4. 사용자가 선택할 수 있도록 짧게 묻습니다.

예시:

```text
지금은 트랙을 먼저 고르면 좋아요.

- crawl: 네이버/에펨코리아/디시/토스 수집
- data: 종목 인식, 감성, 30분 집계
- market: 시세/호가, quote cache
- trade: 가상 계좌, 주문, 체결
- agent: AI 판단, 커뮤니티별 성과 비교
- front: 대시보드와 화면
- ops: 기획, 문서, Notion, PR/CI

현재 추천은 crawl의 parser 보강이나 data의 종목 인식 보강입니다.
어느 트랙으로 갈까요?
```

## 짧은 트랙 지시 처리

사용자가 아래처럼 말하면 충분합니다.

```text
front 작업해
crawl 쪽 맡아줘
ops로 Notion 정리해
agent 트랙에서 커뮤니티별 성과 비교 설계해
```

에이전트는 내부적으로 아래 프롬프트를 확장해 인지합니다.

```text
너는 너나사 (YouBuyFirst)의 <트랙명> 담당 에이전트다.
AGENTS.md, docs/CURRENT_HANDOFF.md, docs/DOCUMENTATION_GUIDE.md를 읽는다.
필요하면 docs/FINAL_PRODUCT_PLAN.md와 담당 트랙 README를 읽는다.
이 채팅에서는 지정된 작업 범위와 해당 트랙만 다룬다.
다른 트랙 파일은 명시 요청이나 계약 변경 필요성이 없으면 건드리지 않는다.
작업 하나는 브랜치 하나와 PR 하나로 만든다.
PR 설명과 작업 기록은 한국어로 작성한다.
```

## 작업 시작 선언

트랙과 범위가 정해지면 에이전트는 한두 문장으로 선언하고 시작합니다.

```text
작업 트랙: front / 수정 대상: 대시보드 화면 / 기록 위치: PR + Notion 작업 로그 / 주요 위험: API 계약 미확정으로 mock 데이터 경계 유지.
```

```text
작업 트랙: ops / 수정 대상: Notion 사용법과 repo 문서 / 기록 위치: PR + Notion 작업 로그 / 주요 위험: child DB 보존과 문서 과밀화.
```

이 선언은 사용자가 직접 써줄 필요가 없습니다. 에이전트가 스스로 추론해서 말합니다.

## 범위가 불명확할 때

에이전트는 바로 구현하지 않고 추정 범위를 먼저 말합니다.

```text
이 요청은 data와 agent가 걸쳐 있습니다. 종목 인식과 감성 분류는 data, 매매 판단은 agent입니다. 먼저 data 계약부터 정리할까요?
```

```text
이건 front 작업처럼 보이지만 API 계약이 아직 없습니다. 화면 mock을 먼저 만들거나, ops/data와 API 응답 계약을 먼저 확인하는 두 방법이 있습니다.
```

## 기획 확인이 필요한 경우

아래 상황에서는 작업자가 단독 구현하지 말고 기획/ops 확인을 요청합니다.

- 트랙 경계를 바꾸는 경우
- 공개 배포 정책, 크롤링 정책, 법적 리스크에 영향을 주는 경우
- DB schema나 API contract가 여러 트랙에 영향을 주는 경우
- 화면 정보 구조가 제품 방향을 바꾸는 경우
- 사용자에게 보이는 핵심 지표 정의를 바꾸는 경우

예시:

```text
이 변경은 crawl 정책과 공개 배포 리스크가 함께 걸립니다. 구현 전에 ops 기획 에이전트에게 수집 정책을 확인하고 오세요.
```

```text
이 화면은 front 작업이지만 지표 정의가 data와 연결됩니다. mock 화면은 진행할 수 있고, 실제 API 연결은 data 계약 확인 후 별도 PR로 나누겠습니다.
```

## Notion 업데이트 책임

Notion은 각 트랙이 자기 기록을 남기고, `ops`가 전체 지도를 관리하는 구조입니다.

| 영역 | 책임자 | 갱신 시점 |
| --- | --- | --- |
| 작업 로그 DB | 각 트랙 에이전트 | PR 생성/수정/머지 시 |
| 다음 작업 DB | 각 트랙 에이전트 + ops | 작업 후보가 생기거나 우선순위가 바뀔 때 |
| 개발자 기술 경험 DB | 문제를 겪은 트랙 에이전트 | 제품 개발/운영 문제, 성능/품질 개선, 기술 결정이 있을 때 |
| 에이전트 운영 로그 DB | 사고를 낸 에이전트 또는 ops | Codex/Notion/GitHub/PR/문서 운영 문제가 생겼을 때 |
| 루트 대시보드 | ops | 주요 PR 머지 후, 트랙 상태가 바뀐 후, 사용자가 점검을 요청했을 때 |
| 트랙별 진행판 | 각 트랙 에이전트가 1차 갱신, ops가 정합성 점검 | 트랙 작업 시작/완료/차단 시 |
| 제품 기획과 작업 맥락 | ops | 제품 방향, 범위, 지표 정의, 트랙 경계가 바뀔 때 |

정리하면, 각 트랙은 자기 작업의 사실 기록을 남기고, ops는 사람이 보는 프로젝트 지도와 기획 정합성을 관리합니다.

## 작업자 응답 틀

작업자가 시작할 때:

```text
작업 트랙: <track> / 수정 대상: <target> / 기록 위치: <records> / 주요 위험: <risk>.
먼저 관련 문서를 확인하고, 이 트랙 범위 안에서만 진행하겠습니다.
```

기획 확인이 필요할 때:

```text
이 작업은 <track A>와 <track B>가 겹칩니다. <이유> 때문에 바로 구현하면 충돌 가능성이 있습니다. 먼저 ops 기획 에이전트에게 <확인할 내용>을 확인하고 오세요.
```

진행 가능하지만 계약이 필요한 때:

```text
<작업>은 진행하겠습니다. 다만 <계약/API/schema/정책>이 아직 없으므로 이번 PR은 <mock/fixture/interface>까지만 하고, 실제 연결은 후속 PR로 나누겠습니다.
```

작업을 거절하거나 보류해야 할 때:

```text
이 요청은 현재 트랙 범위를 벗어납니다. <해당 트랙> 작업으로 분리하거나, ops에서 계약을 먼저 정리한 뒤 진행하는 편이 안전합니다.
```
